## 4. TDD-for-AI 实践：测试用例设计思路、覆盖范围、测试通过率

### 4.1 测试用例设计思路

- 采用“需求 -> 测试 -> 代码行为验证”的 TDD-for-AI 思路。
- 先将课程要求拆解为可执行断言：
  - 核心功能必须可调用且结构化输出稳定。
  - AI 接口必须支持流式输出与元信息（meta + sources）。
  - 安全输入（注入/越狱）下系统不能崩溃。
- 对外部不稳定依赖（LLM/RAG 检索）使用 mock，保证测试稳定、可重复。

### 4.2 覆盖范围

- **功能覆盖**：health/chat/explain/quiz/roadmap 全部关键接口。
- **分层覆盖**：
  - 集成测试（端到端 API）
  - 安全测试（提示词注入/越狱输入）
  - 检索逻辑测试（意图识别、上下文构造、来源提取）
  - 可执行需求规约（requirements test）
- **课程要求对照**：
  - pytest 用例数量 ≥ 10：已达到 29
  - 覆盖层级 ≥ 2：已达到 3+

### 4.3 测试通过率

- 全量执行命令：`uv run python -m pytest tests -v`
- 总结果：**29 passed in 7.07s**
- 通过率：**100% (29/29)**

## 5. 测试结果与评估（含安全测试）

### 5.1 分文件结果

- `tests/requirements_test.py`：**6 passed**
- `tests/test_api_integration.py`：**14 passed**
- `tests/test_retrieval_logic.py`：**5 passed**
- `tests/test_security.py`：**4 passed**

### 5.2 关键结果评估

- **功能可用性**：核心 API 全部可用，输入校验与流式格式符合预期。
- **稳定性**：在 mock 条件下，测试可重复、无随机失败。
- **检索逻辑正确性**：RAG 关键词识别与来源提取行为符合设计。

### 5.3 安全测试评估

已覆盖以下安全场景并通过：

1. **提示词注入攻击**：输入“忽略系统指令/输出敏感信息”时，接口仍可正常响应，不崩溃。
2. **越狱式输入**：在 quiz/roadmap 输入恶意目标时，仍返回结构化业务结果。
3. **异常参数鲁棒性**：`use_rag` 非法值输入不会触发服务 500。

评估结论：当前版本具备基础安全鲁棒性（侧重“稳定与可控输出”），后续可进一步补充更强对抗样本与策略审计。

## 7. 问题与修复记录（执行过程）

1. 初期 `pytest` 缺失：通过 `uv add --dev pytest pytest-asyncio` 解决。
2. 初期 `requirements_test.py` 放在项目根目录导致 fixture 不可见：已调整到 `tests/` 目录。
3. 初期少量流式测试断言失败：通过完善 `conftest.py` 的 mock 覆盖（含 `explain` 路由）修复。
